Pull Request Process

This guide covers how to submit a pull request (PR) and what to expect during the review process.

Before Submittingâ€‹

1. Sync with Upstreamâ€‹

# Fetch latest changes
git fetch upstream

# Rebase your branch
git rebase upstream/main

2. Run Testsâ€‹

# Backend tests
npm test

# Frontend tests
cd frontend-ui && npm test

# Agent tests
cd agents/stella-agent && pytest

3. Check Formattingâ€‹

# Backend
npm run lint
npm run format:check

# Python
black --check agents/
pylint agents/

4. Update Documentationâ€‹

If your changes affect:

  • API endpoints â†’ Update Swagger docs
  • Configuration â†’ Update README or docs
  • New features â†’ Add documentation page

Creating a Pull Requestâ€‹

Branch Namingâ€‹

<type>/<short-description>

Examples:
feature/custom-tts-provider
fix/session-cleanup-error
docs/streaming-guide
refactor/simplify-audio-pipeline

PR Titleâ€‹

Follow Conventional Commits:

feat(agents): add support for ElevenLabs TTS
fix(backend): handle WebSocket disconnection gracefully
docs: add contributing guide

PR Description Templateâ€‹

## Summary

Brief description of the changes.

## Changes

- Added X feature
- Fixed Y bug
- Updated Z documentation

## Testing

Describe how you tested the changes:

- [ ] Unit tests pass
- [ ] Manual testing completed
- [ ] Tested on Kubernetes

## Screenshots (if applicable)

Add screenshots for UI changes.

## Related Issues

Closes #123

Review Processâ€‹

1. Automated Checksâ€‹

When you open a PR, automated checks run:

CheckDescription
CI BuildCompile and build all components
TestsRun unit and integration tests
LintingCheck code style
Type CheckTypeScript/mypy type checking

All checks must pass before merge.

2. Code Reviewâ€‹

A maintainer will review your PR. They may:

  • Approve: Ready to merge
  • Request changes: Need modifications
  • Comment: Questions or suggestions

3. Responding to Feedbackâ€‹

# Make requested changes
git add .
git commit -m "address review feedback"

# Push to update PR
git push

If you need to rebase:

# Rebase on latest main
git fetch upstream
git rebase upstream/main

# Force push (only on your feature branch!)
git push --force-with-lease

4. Mergeâ€‹

Once approved:

  • Maintainer merges the PR
  • Your branch is deleted
  • Changes appear in main

Review Guidelinesâ€‹

If you're reviewing PRs, consider:

Code Qualityâ€‹

  • Code is readable and well-structured
  • Functions are focused and not too long
  • No unnecessary complexity
  • Appropriate error handling

Testingâ€‹

  • New code has tests
  • Tests cover edge cases
  • Tests are readable

Documentationâ€‹

  • Public APIs are documented
  • Complex logic has comments
  • README updated if needed

Securityâ€‹

  • No secrets in code
  • Input validation present
  • No SQL injection risks

Common Issuesâ€‹

CI Failuresâ€‹

# Check CI logs for the specific error

# Common fixes:

# Lint errors
npm run lint:fix

# Test failures
npm test -- --verbose

# Type errors
npx tsc --noEmit

Merge Conflictsâ€‹

# Update your branch
git fetch upstream
git rebase upstream/main

# Resolve conflicts in each file
# Then continue
git add .
git rebase --continue

# Push updated branch
git push --force-with-lease

Large PRsâ€‹

If your PR is large, consider:

  1. Split into smaller PRs: Easier to review
  2. Add context: Explain why it's large
  3. Create an issue first: Discuss the approach

PR Best Practicesâ€‹

Doâ€‹

  • Keep PRs focused on one thing
  • Write clear commit messages
  • Respond to feedback promptly
  • Test your changes thoroughly
  • Update documentation

Don'tâ€‹

  • Mix unrelated changes
  • Force push to shared branches
  • Ignore CI failures
  • Submit without testing
  • Leave stale PRs open

After Mergeâ€‹

  1. Delete your branch: Keep the repo clean
  2. Update local main: git checkout main && git pull
  3. Celebrate: Your contribution is live!

Getting Helpâ€‹

  • Stuck on an issue? Comment on the PR
  • Need clarification? Ask the reviewer
  • CI issues? Check the logs first
