Release Process

This document describes how we version, release, and distribute STELLA.

Versioningâ€‹

We use Semantic Versioning (SemVer):

MAJOR.MINOR.PATCH

Examples:
1.0.0  - First stable release
1.1.0  - New features, backward compatible
1.1.1  - Bug fixes
2.0.0  - Breaking changes

Version Meaningâ€‹

Version ChangeWhen to UseExample
MAJORBreaking API changesRenamed endpoints
MINORNew features (backward compatible)New agent type
PATCHBug fixesFix crash on disconnect

Pre-release Versionsâ€‹

1.0.0-alpha.1  - Early testing
1.0.0-beta.1   - Feature complete, testing
1.0.0-rc.1     - Release candidate

Release Scheduleâ€‹

TypeFrequencyContents
MajorAs neededBreaking changes
MinorMonthlyNew features
PatchAs neededBug fixes, security

Release Checklistâ€‹

1. Prepare Releaseâ€‹

# Ensure on main branch
git checkout main
git pull upstream main

# Create release branch
git checkout -b release/v1.2.0

2. Update Version Numbersâ€‹

# Root package.json
npm version minor

# Frontend
cd frontend-ui && npm version minor && cd ..

# Agents (pyproject.toml or setup.py)
# Update version manually

3. Update Changelogâ€‹

Add entry to CHANGELOG.md:

## [1.2.0] - 2024-01-15

### Added
- New feature X (#123)
- Support for Y (#124)

### Changed
- Improved performance of Z (#125)

### Fixed
- Bug in session cleanup (#126)

### Security
- Updated dependency A to fix CVE-XXXX (#127)

4. Run Full Test Suiteâ€‹

# Backend
npm test
npm run test:e2e

# Frontend
cd frontend-ui && npm test

# Agents
cd agents/stella-agent && pytest
cd agents/stella-light && pytest

5. Create Release PRâ€‹

git add .
git commit -m "chore: prepare release v1.2.0"
git push origin release/v1.2.0

Create PR: release/v1.2.0 â†’ main

6. After PR Mergeâ€‹

# Create and push tag
git checkout main
git pull
git tag -a v1.2.0 -m "Release v1.2.0"
git push upstream v1.2.0

7. Create GitHub Releaseâ€‹

  1. Go to GitHub Releases
  2. Click "Create a new release"
  3. Select tag v1.2.0
  4. Title: "v1.2.0"
  5. Description: Copy from CHANGELOG
  6. Publish release

8. Build and Push Imagesâ€‹

# Backend
docker build -t ghcr.io/c4dhi/stella-backend:1.2.0 .
docker push ghcr.io/c4dhi/stella-backend:1.2.0

# Agents
cd agents/stella-agent
docker build -t ghcr.io/c4dhi/stella-agent:1.2.0 .
docker push ghcr.io/c4dhi/stella-agent:1.2.0

# Tag as latest
docker tag ghcr.io/c4dhi/stella-backend:1.2.0 ghcr.io/c4dhi/stella-backend:latest
docker push ghcr.io/c4dhi/stella-backend:latest

Hotfix Processâ€‹

For urgent bug fixes:

# Create hotfix branch from tag
git checkout -b hotfix/v1.2.1 v1.2.0

# Make fix
git add .
git commit -m "fix: critical bug in X"

# Update version
npm version patch

# Create PR to main
git push origin hotfix/v1.2.1

After merge:

  1. Tag as v1.2.1
  2. Create GitHub release
  3. Build and push images

Docker Image Tagsâ€‹

TagDescription
latestMost recent stable release
1.2.0Specific version
1.2Latest patch of 1.2.x
1Latest minor of 1.x.x
mainLatest from main branch

Documentation Releasesâ€‹

Documentation is deployed automatically:

  • main branch â†’ https://c4dhi.github.io/STELLA_backend/
  • Pull requests â†’ Preview builds (if configured)

Changelog Formatâ€‹

We follow Keep a Changelog:

# Changelog

All notable changes to this project will be documented in this file.

## [Unreleased]

### Added
- New features not yet released

## [1.2.0] - 2024-01-15

### Added
- Feature description (#PR)

### Changed
- Change description (#PR)

### Deprecated
- Deprecated feature (#PR)

### Removed
- Removed feature (#PR)

### Fixed
- Bug fix description (#PR)

### Security
- Security fix description (#PR)

Breaking Changesâ€‹

When introducing breaking changes:

  1. Document clearly in CHANGELOG
  2. Provide migration guide
  3. Deprecate first when possible
  4. Major version bump

Example migration guide:

## Migration Guide: v1.x to v2.0

### Session API Changes

**Before (v1.x):**
```json
POST /api/sessions
{
  "projectId": "..."
}

After (v2.0):

POST /api/v2/sessions
{
  "project_id": "...",
  "agent_config": {}
}

Required Changesâ€‹

  1. Update API endpoints to use /api/v2/
  2. Change projectId to project_id
  3. Add required agent_config field
